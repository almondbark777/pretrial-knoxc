# Phase 9 — Past-Due EM Fees / Show-Cause Letters

> Append-only paper trail. Done 2026-05-31 (Opus 4.8, 1M).

## What this is

The show-cause letters Alex asked to have "on the website." Alex provided the
canonical **`past-due-em-fees` skill** (`past-due-em-fees.skill`) — the logic he'd
"already got figured out" and told me not to recreate. This phase ports that
skill's **methodology** to Go and reuses its **exact memo template**, wiring both
into the live app as a new report.

The skill computes who is 5+ days behind on GPS monitoring fees and emits a filled
past-due memo (`.docx`) per person, split Open/Closed. Its three CSV inputs (GPS 48
Hours, Payments, New Blue Book) are the *same exports the daily importer already
loads* into `raw_gps_48_hours` / `raw_payments` / `raw_blue_book` — so the website
has the data already and "just knows" who's behind, no upload step.

## How it was done (reuse, not recreate)

- **`internal/emfees/emfees.go`** — a faithful, method-for-method port of
  `scripts/generate_memos.py`: ALLIED $8/day · SCRAM $15/day; 5-day threshold;
  inclusive day counting; mid-period device-switch dual-billing (switch day charged
  at both rates); the daily-fee payment-type filter (`gps/allied/scram/ptr, allied/
  scram gps`, excluding install fees / PTR / drug screens); the `byCase or byIdn`
  falsy-fallback for payment matching; Open Pass-1 over the 48-hour file + Closed
  Pass-2 over blue-book-only people with the start-date fallback chain (first
  payment → Released to Hilltop → Referral); the `!!!`/`test` junk filter. Pure
  function (`Compute`) over `[]map[string]string`, fully unit-tested.
- **`internal/emfees/memo.go`** — fills the office's own
  **`assets/memo_template.docx`** (copied verbatim from the skill, `//go:embed`-ed
  into the binary). The template's fields are Word FORMTEXT form fields padded with
  exactly 5 runs of U+2002; `fillClusters` replaces the 35 placeholder runs in
  document order `[Date, Court, Defendant, IDN, Warrant/Docket, GPS Type, Arrearage]`,
  leaving blank fields (e.g. Court) as em-spaces for the officer — exactly like the
  Python `fill_clusters`. Also builds the Open/ + Closed/ memo **zip**.
- **`internal/db/emfees.go`** — feeds the engine from the raw tables and drops
  **tombstoned** people/cases first, so a supervisor-deleted person never generates
  a dunning letter (same importer-proof guarantee as every other view). (Field
  overrides are not yet spliced in — deliberate v1 cut.)
- **`internal/handlers/emfees.go` + `templates/report_emfees.html`** — the report:
  Open/Closed tables with totals + the skill's spot-check highlights (pink = 30+
  days behind / 2-yr span, yellow = 14+ / Referral-only start), a per-row `.docx`
  download, "Download all memos (.zip)", and a CSV summary. Routes:
  - `GET /reports/em-fees` — the report page
  - `GET /reports/em-fees/memo?kind=open|closed&idn=&case=` — one filled memo
  - `GET /reports/em-fees/memos.zip?kind=open|closed|all` — all memos (Open/+Closed/)
  - `GET /export/em-fees.csv` — summary CSV
  Listed first in the Reports hub, tagged "show-cause letters." The old
  Behind-on-GPS report's stale "letters launchpad" note now points here (that report
  is GPS *coverage* compliance, a different thing).

## Parity — proof the port is faithful

`internal/db/emfees_test.go` runs `db.EMFees` against the offline `db/kh222.db` and
asserts it matches, **record-for-record** (count, totals, and every row's
days/owed/paid/behind/type/start-source), a golden file generated by running the
**actual Python skill** on CSVs exported from that same DB (as-of 2026-05-31):

> Open **126** memos, **$85,090.00**; Closed 0; 8 rows skipped (no billable GPS type).

Then a live download was parsed with `python-docx` and the Go-generated memo came
back **field-for-field identical** to the skill's own memo for the same client
(Date, blank Court, Defendant/IDN/Warrant, "ALLIED GPS device", "arrearage is
$272.00"). Plus 11 engine unit tests (rates, threshold, switch math, payment filter,
junk/no-type skips, blue-book fallbacks) and 3 memo-filler tests.

The offline DB is the *old* 337-row 48-hour snapshot (active installs only, all
OPEN) → Closed is legitimately 0 here; live ptr1 (700+ rows, OPEN+CLOSED) will
populate both lists.

### Regenerating the golden (after any intentional methodology change)

The golden (`internal/db/testdata/emfees_golden.json`) is **gitignored** — it carries
IDNs and is derived from the un-committed offline DB; the parity test `Skip`s when
it's absent (same posture as `golden_test.go` vs `kh222.db`). To regenerate:
`python -m venv`, `pip install python-docx openpyxl`, export `raw_gps_48_hours /
raw_payments / raw_blue_book` to CSV with the SharePoint headers, run
`scripts/generate_memos.py --as-of YYYY-MM-DD`, and dump
`compute_open_and_closed(...)` to the golden JSON. (Full one-liners were used this
session; the skill's source is preserved at `internal/emfees/assets/SKILL_REFERENCE.md`.)
