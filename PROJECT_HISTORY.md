## Phase 0 — Inventory (done 2026-05-30)

Cataloged the canonical **v0.82 (SQL-connected)** PTR Client Lookup HTML app
and the legacy Python codebase. Unpacked the embedded bundle
(`python scripts/unpack.py "PTR Client Lookup v0.82 (SQL-connected).html" assets/`)
into three editable JS assets — `8a6913e5-*` (data layer), `5037ee28-*`
(React components), `ebec2ff2-*` (app root). Documented the 38-endpoint
FastAPI surface and the T-SQL→SQLite shim (`queries.py` / `queries_ext.py` /
`sqlite_compat.py` via sqlglot) that the Go rewrite later replaced wholesale.
Migration `001_app_extensions_sqlite.sql` defined the 11 app extension tables
(notes, tags, court_dates, audit_log, violations, saved_searches,
pinned_defendants, user_prefs, reminders, documents, scheduled_check_ins) —
all eventually activated by later phases.

## Phase 1 — Server health check (done 2026-05-30)

Snapshot of ptr1: all services green (`ptr-webapp`, `cloudflared`,
`ptr-import.timer` at ~7:14 ET), DB `integrity_check = ok`, WAL clean.
Row counts at the time: blue_book 3,955 / check_ins 5,000 (at cap) /
payments 2,826 / gps_48_hours 777. **One RED: no backups existed** — closed by

threshold. Diagnostic script: `tools/ptr1_diag.sh`.

## Phase 2 — Parity audit (done 2026-05-30)

**Key finding: the live tool runs the spec verbatim.** The HTML the Python app
served was md5-identical to canonical v0.82
(`fda9d14ae732fc65ac5b385542f47f6e`), so every business-math row of the parity
matrix was green by construction — the only drift surface was the SQL→JS data
feed. Two REDs found there: **R1**, the `/api/lookup_data` feed dropped
GPS switch/relief/waiver columns when absent from the table (killing
switch-aware billing client-side), and **R2**, case tokenizers split on commas
only, mangling space-joined case numbers.

The phase's lasting artifact is the **golden value set** (trackDate
2026-05-30, generated by `tools/parity_ref.py`, used to acceptance-test the Go
compute port): JONES 1704942 (L1, $20, 1 missed initial) · REASONOVER 1426070
(L2, 5 windows/3 missed, $100) · HANCOCK 1704989 (L3, 5/5, $40) · COLLINS
1704603 (closed L2, 1/0, $20) · AGUILAR 1386687 (SCRAM, 41 days +1 inclusive,
$615 owed) · PIETY 1340291 (ALLIED, 33 days, $264) · LUTTRELL 1374859
(418 GPS days, $6,270 owed, −31 surplus days — pins `-ceil`, not `floor`) ·
synthetic switch SCRAM→ALLIED $361 and relief-freeze strict-`<` $150.

## Phase 3 — Fixes (done 2026-05-30)

Both REDs fixed in the Python layer: R2 → `re.split(r"[,\s]+", …)` in
`insert_referral` + `update_defendant`; R1 → `_ls_rows` emits every mapped
header deterministically (absent column → `""`) so the client-side `colFind`
discovery is stable. Yellows deliberately left (dead legacy code or solved by
the rewrite).

## Phase 4 — Go rewrite (done 2026-05-30)

The whole canonical JS business engine ported to `internal/compute`
(computeCheckIns / computePTRFees / computeGPS + helpers), a native-SQLite
data layer (`BuildClients` joining the raw_* tables — no T-SQL, no shim), chi
router, server-rendered `html/template`, single static binary (pure-Go SQLite,
no CGO). Every Phase-2 golden value reproduced exactly; all 26 business-math
matrix rows green, including the load-bearing subtleties: word-boundary
`\bptr\b` payment filter, GPS +1 inclusive days, switch dual-billing,
relief strict-`<`, surplus `-ceil`, EST trackDate, `[,\s]+` case tokens.
Notable correctness work during the port: multi-case defendants became
`map[string][]*Client` with open-preferred reps (`openRep`), `?case=`
filtering threaded into PTR/GPS compute, Behind roster filtered to open reps,
missed-month window spans the full calendar month. Build:
`GOOS=linux GOARCH=amd64 go build ./cmd/server`; deploy = scp + systemd swap
(see `deploy/DEPLOY_GO.md`).

## Phase 5 — Backups (done & proven 2026-05-31)

Closed Phase 1's RED. `deploy/ptr-backup.sh` takes a **WAL-safe online
snapshot** via Python stdlib `sqlite3.Connection.backup()` (no CLI on the
box), runs `PRAGMA integrity_check` (failure deletes the copy and fails the
unit), tars the migrations, prunes past `RETAIN_DAYS=30`.
`ptr-backup.timer` fires daily **11:45 UTC** (~30 min after the import).
Destination is the dedicated 931 GB ext4 drive at `/mnt/backup/ptr`
(UUID `7a6a0d0a-3d21-4492-b90e-ad7a4d8102a2`, fstab `nofail`). Restore proof
recorded on-box 2026-05-31 (integrity ok; row counts matched). Failure
alerting: `ptr-alert@.service` (ntfy push on unit failure) + a daily
`ptr-backup-freshness.timer` that alerts if the newest backup is missing or
>26 h old. Monitoring (Netdata + the app's `/metrics` endpoint + ntfy phone
alerts) was installed in the same push — full detail in
`deploy/MONITORING.md`.

## Phase 7 — Admin & data-entry layer (done 2026-05-30)

The supervisor/correction layer. Migration `003_admin_sqlite.sql`:
`deleted_idns` **tombstones** (whole-person or per-case; filtered inside
`BuildClients` so a delete vanishes from every view **and survives the Sunday
full reload**; flips to a physical raw_* delete via `IMPORTER_RETIRED=true` at
SharePoint cutover, no code change) and `overrides` (supervisor field
corrections applied on read — allow-listed fields: pretrial_level,
referral_date, case_status, gps_type, closed_date, day_adjustment,
supervising_officer, defendant — flagged in the UI, never silent). Officer
CRUD for notes/tags/court dates/reminders/violations. **Every write audited**
in ET (`/admin/audit`). Two auth tiers: allow-list officers write extension
data; `SUPERVISOR_EMAILS` (∩ allow-list) gate delete/restore/override.
Hardening that followed in the same phase: CSRF synchronizer tokens on all
admin POSTs, `COOKIE_SECURE`, security headers, config-driven
`ALLOWED_EMAILS`. On whole-person delete the app purges that person's
extension rows but keeps `audit_log` as the recovery breadcrumb.

## Phase 9 — Past-due EM fees / show-cause letters (done 2026-05-31)

Faithful Go port of the canonical `past-due-em-fees` skill — methodology
ported, **memo reused, not recreated** (`internal/emfees/assets/
memo_template.docx`, embedded, filled via its 35 FORMTEXT placeholders; blank
Court/Case fields left for the officer). Engine: ALLIED $8 / SCRAM $15 per
day, 5-day threshold, inclusive day counting, mid-period device-switch
dual-billing, daily-fee payment-type filter, Open pass over the 48-hour file +
Closed pass over blue-book-only with the start-date fallback chain
(first payment → Hilltop → Referral), junk filter. **Parity proven
record-for-record** against the Python skill on the offline DB (126 Open /
$85,090.00 / 0 Closed) and a generated memo verified field-for-field identical
via python-docx. Served at `/reports/em-fees` (tables + totals + spot-check
highlights), single `.docx` per client, batch zip (`Open/` + `Closed/`
folders), CSV. Reads only raw_* + extension data; tombstoned people never get
dunning letters; overrides are spliced in so corrected GPS type/name reach the
letters. Later additions: per-supervising-officer rollup (branch
`feature/emfees-by-officer`, awaiting review), and 2026-06-10's **letter
selection + history** — per-row checkboxes drive the batch (CSRF-guarded POST,
zip contains exactly the ticked memos), every generated memo is recorded in
`letter_log` (migration 007; one `letters_generated` audit row per batch), and
the report shows each client's **Last letter** date ("never" otherwise).

## Phase 10 — Website data entry (done 2026-05-31)

First step of "data entry moves into the app": officers add defendants
(`/admin/add_defendant`, later fronted by the console intake wizard),
payments, and check-ins from the website. **Hard constraint honored: never
write raw_*** — app-owned tables `added_defendants` / `added_payments` /
`added_check_ins` (migration `004_dataentry_sqlite.sql`, self-provisioned),
**merged into every read path** via `queryMapsIfExists`: `BuildClients`
(profile/rosters/stats/compute), `LookupDatasets` (the bundled tracker), and
`EMFees` (an app-entered GPS payment reduces arrears). So an app-entered
record shows everywhere immediately and survives the Sunday reload; a wrong
add is deletable by a supervisor (tombstones suppress app-added rows too).
Dates normalized at write time (`normInputDate`: ISO → SharePoint-style
`M/D/YYYY`) so app-entered data parses identically to imported data.
Duplicate-IDN adds rejected against roster + added tables. Officer-tier,
CSRF-guarded, every write audited.
