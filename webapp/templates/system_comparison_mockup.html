🐛 Real Data Quality Problems Found in This Export

"Open" has 3 different spellings — Open (500 rows), OPEN (2,241 rows), open (2 rows). Filtering for one misses the others.
2,495 of 3,437 defendants have no Charge Type recorded — 72% of rows are blank. Any report on charges will be incomplete.
Check-In "Type" field has "rickman" as an entry — someone typed an officer name into the wrong field. No validation exists to catch this.
1,585 of 1,682 payments have no officer recorded — 94% of the Payments list is missing "Officer That Collected Payment."

⚠️ Structural Limitations of SharePoint Lists

  • No live totals — must export to Excel to count anything
  • Four separate lists with no enforced relationship between them
  • No automatic violation flagging or overdue alerts
  • No role-based access — every officer sees every defendant
  • GPS 48 Hours list is completely disconnected from the Blue Book

📎 Current State Manual Excel Reports
📊 How to Make the Monthly Supervision Report

1 Export Each of the 4 Lists Separately

Blue Book → Export to Excel. Check Ins → Export to Excel. Payments → Export to Excel. GPS 48 Hours → Export to Excel. That's 4 separate files to download and open.

2 Clean the Status Column (Every Month)

Find & Replace: "OPEN" → "Open," "open" → "Open." Same for Closed variants. Also fix GPS Type: "IN CUSTODY" → "In Custody." This is manual work every time because SharePoint doesn't validate text fields.

3 Build Pivot Tables for Each List

Count open/closed cases by officer. Count check-in types (In Person vs. Phone). Sum payments by type (GPS vs. PTR). None of these cross-reference automatically.

4 Manually Match GPS Cases to Blue Book

VLOOKUP by defendant name or IDN to merge GPS status into the Blue Book sheet. Risky — name spellings vary, some IDNs are missing, and the merge can silently fail.

5 Flag the 1,585 Payment Records Missing Officer

94% of payment rows have no officer recorded. Before presenting the report, either investigate each one or note the data gap — which means the "collected by officer" analysis is essentially unusable.

⏱ Estimated time per report: 3–5 hours · Multiple error-prone manual steps · Data gaps make results unreliable

What Would It Actually Take to Build This?

A system like this is a standard relational web application. The four SharePoint lists become four database tables (Defendants, CheckIns, Payments, GPS) linked by IDN and Case Number — relationships that already exist in your data but can't be enforced in SharePoint. The data from these four exports could seed the initial database directly. The front-end would be a web application accessible from any browser, including mobile, without requiring Microsoft 365 licenses. A local developer or small firm could build and host this for the county — it does not require an enterprise vendor contract.
